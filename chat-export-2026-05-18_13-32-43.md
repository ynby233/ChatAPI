不一样，**有较大功能交集，但不等价**。  
如果你在做一个面向 agent 开发的 mock server，**把它们做成三套独立适配器，最终收敛到统一内部语义层**，这个思路是对的。

我建议你把它们理解成：

- **Chat Completions**：偏“传统聊天补全接口”
- **Responses**：偏“新一代统一推理/工具调用/多模态接口”
- **Anthropic API**：偏“消息驱动 + 工具调用 + thinking/内容块”的另一套语义体系

---

## 1. 先说结论：能不能统一？

### 可以统一的部分
这三套 API 都能抽象成一些共同能力：

- 输入一组消息
- 指定 system/developer/user/assistant 类角色或近似角色
- 生成文本
- 支持流式输出
- 支持工具调用 / function calling（语义相近，但格式不同）
- 支持多轮上下文
- 支持一定程度的结构化输出

所以如果你的 mock server 目标是：

- 让 agent 框架能“像连真实模型一样”跑起来
- 人工配置响应
- 自动化配置场景、脚本、工具调用结果
- 屏蔽供应商差异

那么做统一内部逻辑完全合理。

---

## 2. 但它们不一样，差异主要在哪

---

### A. 数据模型不一样

#### Chat Completions
核心结构一般是：

- `messages: [{role, content}]`
- 输出 `choices[0].message`
- 工具调用在 `tool_calls` / `function_call`

这是最经典的聊天接口抽象。

#### Responses
核心结构更像：

- `input`
- 输出是一个 **response object**
- 内容常常是 **分段/typed items**
- 支持更丰富的输出项：文本、工具调用、推理片段、多模态等

它比 chat completions 更像“统一事件/统一响应容器”。

#### Anthropic
核心结构通常是：

- `messages`
- `system` 单独字段
- `content` 是 **block 列表**
- 输出也往往是 block 列表
- 工具调用是专门的 `tool_use` / `tool_result`

Anthropic 的 block 化程度更强，和 OpenAI chat completions 的“message string + tool_calls”不是一回事。

---

### B. 工具调用语义不一样

这是你做 agent mock 时最需要重视的点。

#### Chat Completions
通常是：

- assistant 返回 `tool_calls`
- client 执行工具
- 再把工具结果作为 `tool` role 消息发回去

#### Responses
通常也是类似闭环，但接口对象更“事件化/typed”
- 可能返回 function/tool call item
- 再用某种 input item 把 tool result 回传

#### Anthropic
通常是：
- assistant content block 中出现 `tool_use`
- 你再发送一个包含 `tool_result` 的消息回去

**本质一样：模型提出调用意图，客户端执行，再把结果喂回去。**  
但**线上的字段结构、消息归属、增量流形态**都不同。

所以你的统一内部模型里，最好不要直接复用某一家原始字段，而应该抽象成：

```ts
type ToolCallIntent = {
  id: string
  name: string
  arguments: unknown
}

type ToolResult = {
  callId: string
 output: unknown
  isError?: boolean
}
```

然后各适配器去映射。

---

### C. “system/developer/instruction”层级不完全一样

这点很容易在 mock server 里被忽略。

#### Chat Completions / Responses
有的体系里会有：
- `system`
- `developer`
- `user`
- `assistant`

不同角色优先级可能不同。

#### Anthropic
更常见的是：
- `system` 单独字段
- `messages` 里主要是 user/assistant

所以你内部统一层最好抽象成：

- `instructions.high`
- `instructions.app`
- `conversation[]`

或者至少：
- `systemInstruction`
- `messages[]`

不要假设所有供应商都原生支持同样的 role 集合。

---

### D. 输出格式能力相似，但机制不同

三者都可以“尽量输出 JSON / schema 化结果”，但机制不完全一样：

- Chat Completions：很多时候依赖 JSON mode / function calling / schema 参数
- Responses：一般更适合做结构化输出与统一响应项
- Anthropic：常借助 tool use、prompt constraint 或特定 structured output 能力

所以如果你的 mock server 需要“按 schema 输出”，建议在内部定义统一能力：

```ts
type OutputConstraint =
  | { type: "text" }
  | { type: "json_object" }
  | { type: "json_schema"; schema: object }
  | { type: "tool_call_only"; allowedTools: string[] }
```

适配器层再决定怎么降级或映射。

---

### E. 流式输出事件模型差异很大

如果你要支持 agent 调试和前端可视化，**流式是大坑**。

#### Chat Completions
常见是 delta 增量：
- `choices[].delta.content`
- `tool_calls` 增量拼接

#### Responses
更像 typed event stream：
- response created
- output item added
- content delta
- tool call delta
- completed

#### Anthropic
也是事件流，但事件类型、block 生命周期、停止原因都不同。

**建议：统一内部不要直接以“文本 token delta”为唯一事件。**  
应该定义更泛化的内部事件模型，例如：

```ts
type AgentStreamEvent =
  | { type: "message_start"; messageId: string }
  | { type: "text_delta"; messageId: string; text: string }
  | { type: "tool_call_start"; callId: string; name: string }
  | { type: "tool_call_delta"; callId: string; argumentsTextDelta: string }
  | { type: "tool_call_end"; callId: string }
  | { type: "message_end"; messageId: string; finishReason?: string }
```

这样三套协议都能映射进来。

---

### F. 状态管理方式不完全一致

有些 API 更偏：
- 每次都提交完整上下文

有些 API 会支持：
- response id
- previous response chaining
- server-side state / trace-like continuation

如果你的 mock server 要做得长期可扩展，内部最好显式区分：

- **stateless mode**：调用方每次传完整上下文
- **stateful mode**：服务端保存 session / run / response chain

这样之后不管接 Anthropic 还是 Responses，都更灵活。

## 3. 如果你的目标是“agent mock server”，我建议的架构

我非常建议你采用：

## 外层：三套协议适配器
- `adapter/anthropic`
- `adapter/responses`
- `adapter/chat-completions`

职责：
- 请求解析
- 供应商字段校验
- 协议级流式编码/解码
- 内部模型 <-> 外部协议映射
- 外部错误码兼容

---

## 中层：统一内部语义层
比如定义这些核心对象：

### Request
```ts
type UnifiedModelRequest = {
  model?: string
  instructions?: {
    system?: string
    developer?: string
  }
  messages: UnifiedMessage[]
  tools?: UnifiedTool[]
  toolChoice?: UnifiedToolChoice
  outputConstraint?: OutputConstraint
  stream?: boolean
  temperature?: number
  maxOutputTokens?: number
  metadata?: Record<string, unknown>
}
```

### Message
```ts
type UnifiedMessage = {
  role: "user" | "assistant" | "system" | "developer" | "tool"
  content: UnifiedContentPart[]
}
```

### Content part
```ts
type UnifiedContentPart =
  | { type: "text"; text: string }
  | { type: "image"; url?: string; base64?: string; mediaType?: string }
  | { type: "tool_call"; id: string; name: string; arguments: unknown }
  | { type: "tool_result"; callId: string; result: unknown; isError?: boolean }
  | { type: "thinking"; text: string; hidden?: boolean }
```

### Response
```ts
type UnifiedModelResponse = {
  id: string
  output: UnifiedMessage[]
  finishReason?: "stop" | "length" | "tool_call" | "content_filter" | "error"
  usage?: {
    inputTokens?: number
    outputTokens?: number
    reasoningTokens?: number
  }
}
```

## 内核：Mock 执行引擎
内部逻辑可以支持几种模式：

### 1）静态规则模式
根据：
- model
- 最后一条 user message
- tools
- 会话状态

匹配预设响应。

适合人工模拟。

---

### 2）脚本模式
像状态机一样：

- 第 1 轮回复文本
- 第 2 轮触发工具调用 `search_flights`
- 收到 tool result 后再回复
- 第 4 轮故意返回 malformed JSON

适合自动化测试 agent。

---

### 3）策略模式
可配置：
- 总是优先调用工具
- 遇到某关键词输出结构化 JSON
- 随机注入延迟/错误/截断/重试

适合鲁棒性测试。

---

## 4. 你不应该追求“完全功能对齐”，而应该追求“能力分层”

这是关键。

不要试图说：
> “三套 API 的每个字段我都统一成一个字段”

这会很痛苦，也很脆弱。

建议你分三层能力：

### Level 1：共同子集
所有适配器都支持：
- 文本对话
- 多轮消息
- 流式文本
- 单步工具调用
- 工具结果回传

### Level 2：扩展能力
部分适配器支持：
- 多模态输入
- 严格 JSON schema
- reasoning/thinking
- server-side state
- 并行工具调用

### Level 3：供应商特性直通
例如：
- Anthropic 特有 block/event
- Responses 特有 response item/event
- Chat Completions 某些历史字段

你可以在统一层提供 `extensions` 字段保底：

```ts
type UnifiedExtensions = {
  vendor?: "anthropic" | "openai-responses" | "openai-chat"
  rawRequest?: unknown
  rawResponse?: unknown
  vendorSpecific?: Record<string, unknown>
}
```

这样你不会因为“过度统一”而丢失重要语义。

## 5. 实际设计上，Anthropic 和 Chat Completions 不建议直接互转，应先转 IR

也就是中间表示 IR（intermediate representation）。

### 不推荐
- Anthropic request 直接转 chat completions request
- Responses 直接转 anthropic request

因为会出现：
- role 丢失
- content block 丢失
- tool use 事件不完整
- stream event 无法一一对应

### 推荐
- 外部协议 A -> 统一 IR
- IR -> Mock 引擎
- Mock 引擎输出 IR
- IR -> 外部协议 B/C

这也是你现在想做的“独立适配器模块 + 统一内部逻辑模块”的最佳实践。

---

## 6. 对 agent mock server 来说，最值得先支持的最小闭环

如果你想优先落地，我建议先只做这几个能力：

### 必做
- 多轮 messages
- system 指令
- 文本输出
- streaming
- 单个 tool call
- tool result 回传
- configurable finish reason
- token usage mock

### 第二阶段
- 多个 tool calls
- 并行 tool calls
- JSON schema 输出
- 图片输入
- 延迟/超时/错误注入
- 截断流 / 半包流 / 非法事件

### 第三阶段
- thinking/reasoning 模拟
- server-side state
- response chaining
- prompt caching / cache 标记
- 更真实的 usage 统计

## 7. 一个很实用的统一抽象建议

如果你问我“统一内部逻辑最核心该围绕什么设计”，我的答案是：

不要围绕“chat completion 文本生成”设计，  
而要围绕 **agent step** 设计。

例如：

```ts
type AgentStep =
  | {
      type: "assistant_message"
      content: UnifiedContentPart[]
    }
  | {
      type: "tool_call"
      calls: ToolCallIntent[]
    }
  | {
      type: "tool_result_request"
      pendingCalls: string[]
    }
  | {
      type: "final"
      content: UnifiedContentPart[]
      finishReason: string
    }
```

因为 agent 场景里，模型本质不是只“回一句话”，而是在不断地产生：

- 文本
- 决策
- 工具调用
- 等待外部执行
- 再整合结果

这比把内部逻辑建立在“assistant message string”上更稳。
